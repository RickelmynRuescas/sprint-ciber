// 01 · Autenticação

Identidade verificada por token assinado.

JWT com algoritmo HS256, expiração de 1 hora e chave lida de variável de ambiente. Cada requisição passa por filtro dedicado antes de chegar ao controller.

Client
→

→

→

→

O filtro executa em toda requisição — antes de qualquer controller. Token ausente ou inválido: request bloqueada silenciosamente, sem revelar informação interna.

JwtService.java · gerarToken() 
// Token inclui role, subject, emissão e expiração
return Jwts.builder()
  .setClaims(Map.of("role", role))
  .setSubject(nomeUsuario)
  .setIssuedAt(new Date(System.currentTimeMillis()))
  .setExpiration(new Date(System.currentTimeMillis() + 1000 * 60 * 60)) // 1 hora
  .signWith(getChaveDeEntrada(), SignatureAlgorithm.HS256)
  .compact()

// Chave derivada da variável de ambiente ${SECRET_KEY_JWT}
private Key getChaveDeEntrada() {
    return Keys.hmacShaKeyFor(secretKey.getBytes());
}

// 1. Extrai e valida header Authorization
final String header = request.getHeader("Authorization");
if (header == null || !header.startsWith("Bearer ")) {
    filterChain.doFilter(request, response);
    return; // sem token → passa sem autenticação

// 2. Valida token e carrega contexto de segurança

    var auth = new UsernamePasswordAuthenticationToken(

    );
    SecurityContextHolder.getContext().setAuthentication(auth);

// 02 · Validação de Entrada

Nenhum dado entra sem ser inspecionado.

Bean Validation do Jakarta EE valida cada campo antes de chegar à camada de serviço. Erros retornam em formato padronizado — sem stack trace, sem informação interna.

@NotBlank Campo obrigatório — rejeita null, vazio e só espaços em branco OBRIGATÓRIO
@Email Formato de e-mail validado pelo Jakarta antes de persistir FORMATO
@Pattern CPF: exatamente 11 dígitos numéricos via regex ^\d{11}$ REGEX
@NotNull Referências obrigatórias — clienteId, categoriaServico, tipoFalha, veiculo REFERÊNCIA
@Valid Cascata de validação — aplica regras em objetos aninhados como VeiculoDTO CASCATA

// 03 · Proteção de API

Cada endpoint protegido por role.

Spring Security com RBAC define exatamente quem acessa o quê. Sessão stateless elimina ataques de fixação de sessão. CSRF desabilitado — correto para APIs JWT.

/api/v1/clientes/** GET · PUT · DEL AUTH

SecurityConfig.java · filterChain() 
http
  .csrf(csrf -> csrf.disable())                               // JWT = stateless, CSRF desnecessário
  .sessionManagement(sm -> sm
    .sessionCreationPolicy(SessionCreationPolicy.STATELESS))   // sem sessão HTTP
  .authorizeHttpRequests(auth -> auth
    .requestMatchers(POST, "/api/v1/auth/login").permitAll()
    .requestMatchers(POST, "/api/v1/clientes").hasRole("FUNCIONARIO")
    .requestMatchers(POST, "/api/v1/ordens-servicos").hasRole("FUNCIONARIO")
    .requestMatchers(POST, "/api/v1/usuarios").hasRole("FUNCIONARIO")
    .anyRequest().authenticated()                              // tudo mais exige token
  )
  .addFilterBefore(jwtFilter, UsernamePasswordAuthenticationFilter.class);

Princípio do Menor Privilégio

Criação de dados exige FUNCIONARIO. Leitura e atualização requerem apenas token válido.

CSRF Desabilitado Corretamente

APIs stateless com JWT não precisam de proteção CSRF — o token já é a prova de identidade por request.

Filter Before Auth

JwtAuthFilter executa antes do filtro padrão do Spring — intercepta e valida antes de qualquer processamento.
